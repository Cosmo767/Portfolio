What is A/B Testing?

A/B testing (also called split testing) is a method of comparing two versions of something to determine which performs better. You show version A to one group of users and version B to another group, then measure which version achieves your goal more effectively.

Real-World Example: An e-commerce company wants to know if changing their "Buy Now" button from blue to green increases purchases. They randomly show 50% of visitors the blue button (Control) and 50% the green button (Variant), then compare conversion rates.

Key Components

Component Description
Control (A) The current version (baseline)
Variant (B) The new version you're testing
Metric What you're measuring (conversion rate, clicks, revenue, etc.)
Sample Size Number of users in each group
Statistical Significance Confidence that results aren't due to chance (usually 95%)

When Should You Run an A/B Test?

✅ Good Use Cases

  • Testing different button colors, sizes, or copy
  • Comparing two email subject lines
  • Evaluating different page layouts
  • Testing pricing strategies
  • Optimizing checkout flows

❌ When NOT to A/B Test

  • Not enough traffic (need sufficient sample size)
  • Testing completely different products (use separate launches)
  • When stakeholders have already decided (testing should inform decisions)
  • Testing too many variables at once (causes statistical issues)
⚠️ Minimum Traffic Required: You typically need at least 1,000 visitors per variant to detect meaningful differences. With lower traffic, tests take too long or can't reach statistical significance.

Common Pitfalls to Avoid

1. Stopping Tests Too Early

"Peeking" at results and stopping when you see significance leads to false positives. Always run tests for the pre-calculated duration.

Example: You calculate you need 10,000 visitors per variant. On day 3, you have 3,000 visitors and see the variant is winning with p < 0.05. If you stop now, there's a high chance this is a false positive. Wait for 10,000.

2. Testing Too Many Things at Once

Testing multiple changes simultaneously makes it impossible to know what caused the difference. Test one variable at a time.

3. Not Accounting for Seasonality

Running tests during holidays, sales events, or other special periods can skew results. Ensure your test runs through at least one full business cycle.

4. Ignoring Statistical Power

Low statistical power means you might miss real improvements. Always calculate required sample size before starting.

5. Not Randomizing Properly

Ensure users are randomly assigned to groups. Don't let users choose their variant or assign based on non-random factors.

Best Practices Checklist

Before the Test

  • ☐ Define clear hypothesis and success metric
  • ☐ Calculate required sample size
  • ☐ Set significance level (usually 95%)
  • ☐ Ensure proper randomization
  • ☐ Test only one variable

During the Test

  • ☐ Don't peek at results early
  • ☐ Monitor for implementation bugs
  • ☐ Ensure traffic is split evenly
  • ☐ Run for full calculated duration

After the Test

  • ☐ Calculate statistical significance
  • ☐ Consider practical significance (is the improvement meaningful?)
  • ☐ Document results and learnings
  • ☐ Roll out winner to 100% of users
  • ☐ Monitor for long-term effects
Pro Tip: Keep a testing log with all your hypotheses, results, and learnings. Failed tests are just as valuable as winners – they tell you what doesn't work and prevent others from testing the same thing.

Bonus: Python Code for A/B Testing

Here's how to run these calculations in Python:

from scipy import stats import numpy as np # Sample Size Calculation def calculate_sample_size(baseline_rate, mde, alpha=0.05, power=0.8): """Calculate required sample size per variant""" p1 = baseline_rate p2 = baseline_rate * (1 + mde) # Z-scores for alpha and power z_alpha = stats.norm.ppf(1 - alpha/2) z_beta = stats.norm.ppf(power) # Pooled probability p_pooled = (p1 + p2) / 2 # Sample size formula n = (2 * p_pooled * (1 - p_pooled) * (z_alpha + z_beta)**2) / (p1 - p2)**2 return int(np.ceil(n)) # Statistical Significance Test def ab_test_significance(control_visitors, control_conversions, variant_visitors, variant_conversions): """Perform two-proportion z-test""" # Calculate conversion rates p1 = control_conversions / control_visitors p2 = variant_conversions / variant_visitors # Pooled proportion p_pooled = (control_conversions + variant_conversions) / (control_visitors + variant_visitors) # Standard error se = np.sqrt(p_pooled * (1 - p_pooled) * (1/control_visitors + 1/variant_visitors)) # Z-statistic z_stat = (p2 - p1) / se # P-value (two-tailed) p_value = 2 * (1 - stats.norm.cdf(abs(z_stat))) return { 'control_rate': p1, 'variant_rate': p2, 'improvement': (p2 - p1) / p1, 'p_value': p_value, 'significant': p_value < 0.05 } # Example usage sample_size = calculate_sample_size(0.05, 0.10) print(f"Required sample size: {sample_size} per variant") results = ab_test_significance(10000, 500, 10000, 580) print(f"Control rate: {results['control_rate']:.2%}") print(f"Variant rate: {results['variant_rate']:.2%}") print(f"Improvement: {results['improvement']:.2%}") print(f"P-value: {results['p_value']:.4f}") print(f"Significant: {results['significant']}")
